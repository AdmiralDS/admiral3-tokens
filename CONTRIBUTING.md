# Contributing

В этом документе зафиксированы обязательные правила внесения изменений в репозиторий.

## Коммиты

Для сообщений коммитов используется `Conventional Commits`:

https://www.conventionalcommits.org/ru/v1.0.0/

### Формат сообщения

Базовый формат:

```text
type(scope): short description
```

Примеры:

```text
feat(tokens): add storybook playground template
fix(testing): correct vitest typings for jest-dom matchers
docs(contributing): describe pull request preparation flow
chore(ci): align storybook and e2e checks
```

### Обязательные правила для коммитов

1. Название коммита должно точно соответствовать сути изменений.
2. Один коммит должен решать одну логически завершённую задачу.
3. В коммит должны попадать только файлы, которые действительно относятся к этой задаче.
4. Нельзя смешивать в одном коммите разные типы изменений без необходимости.

Плохие примеры:

- одновременно изменение токенов, тестов, документации и CI без общей причины;
- исправление линта в случайных несвязанных файлах "заодно";
- добавление инфраструктурных правок в коммит, который называется как изменение одной группы токенов.

Хорошая практика:

- `feat(...)` только для новой функциональности;
- `fix(...)` только для исправления дефекта;
- `docs(...)` только для документации;
- `test(...)` только для тестового контура;
- `chore(...)` для технических изменений без пользовательской функциональности.

### Требования к содержанию коммита

Каждый коммит должен быть:

1. Рабочим.
2. Самодостаточным.
3. Проверяемым.

Это означает:

1. Коммит не должен оставлять проект в заведомо сломанном состоянии.
2. После коммита код должен собираться и проходить релевантные проверки для изменённой области.
3. Нельзя коммитить промежуточное состояние, которое "починится в следующем коммите", если этого можно избежать.
4. Если изменение требует обновления тестов, stories или документации, они должны входить в тот же коммит, если относятся к одной задаче.

### Чего делать нельзя

1. Делать коммит с названием, которое не отражает фактические изменения.
2. Коммитить несвязанные правки "пакетом", чтобы сократить число коммитов.
3. Оставлять в коммите неиспользуемые файлы, временные правки, закомментированный мусор или случайные форматирования не по теме.
4. Делать коммит, если локально не пройдены обязательные проверки для этой задачи.

## Подготовка изменений перед PR

Перед созданием pull request изменения нужно привести в чистое состояние и прогнать обязательные проверки.

CI запускается на Node.js 24. Для локальной проверки используйте ту же major-версию Node.js и устанавливайте зависимости через `npm ci`.

### Команды

Основные локальные потоки:

```shell
npm run check:fix
npm run check:full
npm run storybook
npm run playground
npm run release
```

Автоматическое исправление форматирования и линта:

```shell
npm run check:fix
```

Полный прогон обязательных проверок:

```shell
npm run check:full
```

В `npm run check:full` сейчас входят:

1. `npm run format:check`
2. `npm run lint`
3. `npm run typecheck`
4. `npm run test:run`
5. `npm run test:e2e`

`npm run storybook` запускает Storybook из исходников и не требует предварительной сборки CSS.
`npm run playground` генерирует CSS-токены и запускает Vite playground с hot reload.

Подробные правила по организации unit-тестов, e2e и Storybook stories вынесены в [tests/TESTING_README.md](tests/TESTING_README.md).

### Импорты

Порядок импортов поддерживается автоматически через ESLint.

Что это означает:

1. Не нужно вручную переставлять import-блоки "на глаз".
2. Если порядок импортов сбился, нужно запускать `npm run lint:fix` или `npm run check:fix`.
3. В PR не должно оставаться файлов с импортами, которые расходятся с автосортировкой линтера.

### Обязательный порядок перед PR

1. Убедиться, что изменения логически собраны и разбиты на корректные коммиты.
2. Запустить `npm run check:fix`, если нужны автоисправления форматирования и линта.
3. Запустить `npm run check:full`.
4. Убедиться, что PR содержит только файлы, относящиеся к задаче.
5. Проверить, что описание PR соответствует фактическим изменениям.

### Что обязательно проверить вручную

Перед PR нужно убедиться в следующем:

1. Изменение действительно решает заявленную задачу.
2. В PR нет случайных файлов и побочных правок.
3. Название PR и названия коммитов отражают фактическое содержание.
4. Все обязательные проверки завершились успешно.
5. Если менялись токены, обновлены связанные tests, stories, playground-сценарии и документация, когда это необходимо.

Если PR затрагивает тестовую структуру или stories, нужно дополнительно проверить соответствие правилам из [tests/TESTING_README.md](tests/TESTING_README.md).

## Release и publish

Публикация пакета завязана на GitHub Release.

### Текущий релизный поток

1. Убедиться, что рабочее дерево чистое и изменения уже в `main`.
2. Локально прогнать `npm run check:full`.
3. Подготовить релиз через `npm run release`.
4. Проверить созданные изменения версии и changelog.
5. Запушить commit и tag командой `git push origin main --follow-tags`.
6. Создать GitHub Release по этому tag.
7. Дождаться workflow [`.github/workflows/npm_release.yaml`](.github/workflows/npm_release.yaml), который:
   - поднимается на событие `release.published`;
   - повторно запускает format, lint, unit, e2e, build;
   - проверяет tarball через `npm pack --dry-run`;
   - публикует пакет в npm только после успешной верификации.

### Что важно

1. `standard-version` по умолчанию создаёт tag с префиксом `v`, и GitHub Release должен создаваться по нему.
2. Публикация выполняется из чистого runner и всегда пересобирает `dist`, поэтому release не зависит от локально собранных файлов.
3. Если нужен повторный запуск после инфраструктурного сбоя, workflow можно перезапустить через `workflow_dispatch`, но штатный путь публикации остаётся через GitHub Release.

## Accessibility

Accessibility является обязательной частью качества демонстрационных сценариев, а не дополнительной опцией "на потом".

### Базовое правило

Все новые stories должны проходить проверки `@storybook/addon-a11y`.

В проекте Storybook accessibility включён в строгом режиме. Это означает:

1. Accessibility violations в stories считаются ошибкой.
2. Новую или изменённую story нельзя считать готовой, если она падает по a11y.
3. Исправление accessibility не должно откладываться в отдельную задачу без явной причины.

### Что проверяется автоматически

Storybook accessibility checks помогают ловить, в частности:

1. Проблемы с ролями и семантикой.
2. Отсутствующие или некорректные label/aria-атрибуты.
3. Частые ошибки структуры markup.
4. Часть проблем с визуальной доступностью и контрастом.

### Что остаётся обязанностью автора изменений

Автоматические проверки не заменяют инженерное решение. Автор изменений обязан проверить:

1. Корректную семантику демонстрационных примеров.
2. Работу с клавиатуры, если пример содержит интерактивные controls.
3. Видимый focus-state.
4. Корректное поведение состояний, если они показываются в примере.
5. Понятный и безопасный пример использования токенов без скрытых accessibility-требований.

### Минимальный accessibility DoD

Для любой новой или изменённой story минимум включает:

1. Story показывает реальный сценарий использования токенов.
2. Story проходит Storybook accessibility checks без violations.
3. Если пример интерактивный, поведение с клавиатуры и focus покрыто тестом или явно проверено вручную.
4. Если для доступного использования токена есть ограничения, они отражены в документации или рядом с примером.

## Новые токены

Этот пакет не предназначен для добавления новых UI-компонентов. Основной тип изменений здесь - добавление и сопровождение design tokens: colors, typography, radius, shadow, а также будущих групп вроде `animation`, `breakpoints`, `zIndex`.

### Общий порядок добавления

1. Определить, к какой группе относится токен: существующей (`color`, `typography`, `radius`, `shadow`) или новой.
2. Добавить source of truth в `src/tokens/<group>`, не дублируя значения в unrelated-файлах.
3. Добавить локальный `index.ts` для новой группы или обновить существующий barrel.
4. Если токен входит в theme object, подключить его в `src/tokens/themes/buildTheme.ts` и обновить theme types.
5. Если токен должен быть доступен как CSS custom property, подключить его в `src/css-data.ts` и `scripts/build-css.ts`.
6. Экспортировать публичный TypeScript API через `src/index.ts`, если токен должен быть доступен из root entrypoint.
7. Обновить `package.json` exports только если появляется новый публичный subpath, например `./css/animation`.
8. Добавить или обновить unit-тесты на форму данных, aliases, theme values и CSS generation.
9. Добавить Storybook scenario для визуальной проверки, если токен имеет визуальное представление.
10. Добавить сценарий в internal playground только если этот кейс нужно покрывать через e2e или проверять как browser/runtime integration.
11. Обновить `README.md`, `PROJECT-MAP.md` и `CONTRIBUTING.md`, если меняется публичный API, структура проекта или порядок работы.
12. Запустить релевантные проверки.

### Структура новой группы токенов

Минимальная структура для новой группы:

```text
src/tokens/<group>/
  index.ts
  <group>.ts
```

Если у группы есть визуальная демонстрация, по умолчанию story кладётся рядом с этой группой:

```text
src/tokens/<group>/stories/
  <Group>.stories.tsx
  <Group>.template.tsx
  <Group>.args.ts
playground/scenarios/<group>.tsx
```

Это не жёсткое правило для всех stories. В текущей структуре:

1. `shadow` и `typography` stories лежат рядом со своими группами: `src/tokens/shadow/stories` и `src/tokens/typography/stories`.
2. Общие token usage stories и сценарии, которые затрагивают несколько групп или показывают цветовые токены в broader context, лежат в `src/tokens/stories`.
3. Для вложенных групп можно держать stories рядом с конкретной source-группой, если это делает структуру понятнее, например `src/tokens/color/globalColors/stories`.

Новая story должна размещаться там, где она ближе к source of truth и проще поддерживается. Если сценарий относится к одной группе токенов, предпочитайте локальную папку `stories`. Если сценарий объединяет несколько групп или показывает consumer usage, используйте `src/tokens/stories`.

Если группа влияет на CSS output, должны быть обновлены:

```text
src/css-data.ts
scripts/build-css.ts
scripts/build-css.test.ts
```

Если группа входит в theme objects, должны быть обновлены:

```text
src/tokens/themes/buildTheme.ts
src/tokens/themes/buildTheme.test.ts
```

Если изменение добавляет только демонстрационный сценарий для уже существующей группы токенов, source of truth и CSS/theme
генераторы менять не нужно. В этом случае достаточно обновить соответствующие Storybook stories, а playground-сценарии,
e2e-проверки и документацию по структуре проекта обновлять только тогда, когда новый пример должен покрываться через
browser/runtime integration.

### Правила для source of truth

1. Значения токенов должны жить в одном очевидном source-файле внутри `src/tokens/<group>`.
2. CSS, theme objects, stories и playground должны брать данные из source of truth или публичного API, а не копировать значения вручную.
3. Если нужны разные форматы вывода, они должны генерироваться или собираться из одной структуры.
4. Generated files и `dist` не редактируются вручную и не считаются source of truth.
5. Имена токенов должны быть стабильными, семантическими и согласованными с соседними группами.

### Правила для API и экспортов

1. Публичный TypeScript API пакета собирается через `src/index.ts`.
2. У каждой публичной группы токенов должен быть локальный barrel `src/tokens/<group>/index.ts`.
3. Deep imports из `src/tokens/<group>/<file>` не считаются публичным API.
4. Root API должен экспортировать потребительские агрегаты и helpers, а не каждую внутреннюю вложенную token map только ради удобства stories/templates.
5. Consumer-like imports в templates используются только для примеров публичного API. Если template рендерит внутренние source maps или story-only данные, используйте локальные импорты из source-файлов.
6. Новый CSS subpath добавляется только если есть самостоятельный consumer use case для отдельного CSS-файла.
7. Публичные CSS custom properties всегда используют префикс `--admiral-`.
8. Переименование публичных token keys, CSS variables или subpaths считается breaking change.

### Правила для CSS tokens

1. CSS variables должны строиться из структурированных токенов через `scripts/build-css.ts`.
2. Новая CSS-группа должна быть добавлена в список generated files и покрыта тестом.
3. Если токен ссылается на другой токен, предпочтительно сохранять связь через `var(..., fallback)`, как это сделано для semantic colors и typography text styles.
4. `dist/css/index.css` должен включать новую группу только если это ожидаемый базовый импорт для потребителей.
5. Отдельный CSS-файл должен иметь понятное имя и matching subpath в `package.json`, если он публикуется.

### Правила для theme tokens

1. Theme-dependent значения должны явно поддерживать все текущие modes: `light`, `dark`, `lightNeutral`, `darkNeutral`.
2. Нельзя добавлять новый theme mode без отдельного решения по публичному контракту.
3. Если новый токен входит в `BuiltTheme`, нужно обновить типы и тесты `buildTheme`.
4. Строковые aliases добавляются только если они нужны для совместимости с ожидаемым стилем потребления токенов.
5. Изменения публичного поведения тем должны отражаться в Storybook `Themes`, README и, если поведение нужно проверить в браузере, в internal playground/e2e.

### Правила для typography tokens

1. Новые text styles добавляются в `src/tokens/typography/textStyles.ts` и в явную alias-карту `typographyTextStyles`.
2. Если style появляется в Storybook typography, добавлять аналогичный сценарий в internal playground нужно только при наличии e2e/runtime use case.
3. Font family tokens должны оставаться в `src/tokens/typography/primitives.ts`; font assets подключаются отдельно и не должны смешиваться с описанием text styles.

### Примеры будущих групп

`animation`:

1. Хранить duration, easing и semantic motion tokens в `src/tokens/animation`.
2. Для CSS output использовать переменные вида `--admiral-animation-*`.
3. Если animation tokens не зависят от темы, не добавлять их в theme objects без явной необходимости.

`breakpoints`:

1. Хранить значения в `src/tokens/breakpoints`.
2. Экспортировать plain object и типы для TypeScript use cases.
3. CSS variables добавлять только если они реально нужны потребителям; media queries не могут напрямую читать CSS custom properties во всех сценариях, это нужно учитывать в документации.

`zIndex`:

1. Хранить semantic layer names в `src/tokens/zIndex`.
2. Не добавлять случайные числовые значения под один кейс; слой должен описывать повторяемую роль.
3. CSS variables использовать с именами вида `--admiral-z-index-*`.

### Правила для Storybook и internal playground

1. `*.stories.tsx` должен оставаться тонким.
2. Storybook-специфика (`Meta`, `StoryObj`, `args`, `argTypes`, story exports) должна находиться в `*.stories.tsx`.
3. `*.template.tsx` должен содержать только React-код и не должен зависеть от типов или объектов Storybook.
4. `meta` должен описывать общий Storybook config, а `render` должен задаваться явно в каждом story export.
5. Если story показывает код в Docs, `parameters.docs.source.code` должен брать строку из соответствующего `*.template.tsx` через `?raw`, а не полагаться на автогенерацию Storybook.
6. Stories должны использовать публичный импорт пакета, то есть так, как токены будут использоваться потребителем.
7. Stories одной конкретной группы можно хранить рядом с группой; общие usage stories можно хранить в `src/tokens/stories`.
8. Порядок верхнеуровневых разделов Storybook задаётся в `.storybook/preview.tsx` через `parameters.options.storySort.order`.
9. E2E не должны зависеть от Storybook runtime.
10. Сценарии playground должны описываться в `playground/scenarios` и добавляться только для кейсов, которые осмысленно проверять через e2e или browser/runtime integration.
11. `playground/scenarios/index.ts` должен оставаться агрегатором, а не большим монолитным списком всех сценариев вручную.
12. Для новой группы токенов или группы e2e-кейсов нужно добавлять отдельный файл сценариев, например `playground/scenarios/animation.tsx`.
13. Сценарии, которые зависят от текущей `styled-components` темы, должны использовать общий переключатель темы playground, если им не нужен отдельный локальный control; выбранная тема должна сохраняться при переключении сценариев.
14. Интерактивные token stories должны использовать доступные нативные элементы управления или явно заданные ARIA-роли.
15. Theme Colors templates должны использовать общий source-driven renderer, чтобы группы и states брались из token source, а не дублировались вручную в story.
16. Storybook templates и playground UI должны строить цвета, типографику, радиусы и тени из token API, theme object или generated CSS variables. Не добавляйте локальные hex/rgb-цвета, ручные font-family/font-size/line-height/font-weight или фиксированные `border-radius`, если соответствующее значение уже есть в токенах.
17. Для демонстрации global palette или генератора палитр исходные цветовые значения тоже нужно брать из token source, когда пример не имитирует внешний consumer input.

### Минимальный Definition of Done для новых токенов

Токены считаются готовыми к PR, когда выполняются все условия:

1. Source of truth добавлен или обновлён в `src/tokens/<group>`.
2. Публичные exports обновлены только там, где это нужно для consumer API.
3. CSS generation обновлена и покрыта тестом, если токен публикуется как CSS variable.
4. Theme builder и tests обновлены, если токен входит в theme object.
5. Storybook обновлён для новых визуальных сценариев, а playground/e2e обновлены только для сценариев, которые нужно проверять как browser/runtime integration.
6. README, PROJECT-MAP и CONTRIBUTING обновлены, если изменился публичный контракт или структура.
7. Все локальные проверки для изменённой области проходят.

## Минимальное ожидание от каждого PR

Каждый PR должен быть:

1. Ограничен одной задачей или одной группой тесно связанных изменений.
2. Читаем по истории коммитов.
3. Проверяем локально.
4. Готов к ревью без необходимости "доделать потом".

PR не должен использоваться как контейнер для незавершённой или грязной работы, если это не оговорено отдельно.
